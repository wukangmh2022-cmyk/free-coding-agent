# 高频定时任务与微信通知 TODO

## 背景

当前定时计划完成后，会由程序统一把最终结论发送到微信。这适合低频、每次都需要通知的任务，例如日报、吃药提醒。

对于高频监测任务，例如“每 1 分钟检查一次，只有异常才通知微信”，不能只依赖完成态自动通知。Agent 或稳定脚本需要拥有是否通知的决策权。

## 待办

1. 终端扩展指令补齐
   - 增加 `wx send_text 文本`，用于异常或关键事件时主动发微信文字。
   - 保留 `wx send_file 文件路径1,文件路径2`，用于发送产物文件。
   - 没有显式 `wx send_text/send_file` 时，高频任务默认静默。
   - 不把脚本 stdout 解析作为主方案；脚本里是否触发通知应该由脚本真实执行到对应分支决定。
   - 推荐实现为本地后端接口或小 CLI：`wx send_text ...` / `wx send_file ...` 在执行前展开成对 Agent Qt 本地 HTTP 接口的调用，类似终端日志查看接口。
   - 这样脚本可直接在条件分支中调用通知命令，避免普通日志和控制协议混杂。

2. Schedule 通知策略
   - 低频提醒类 schedule 可继续使用完成态自动通知。
   - 监测类 schedule 支持“仅显式 wx 指令时通知”。
   - 计划创建时保存通知目标，微信发起的计划默认带微信目标；桌面发起但用户要求发微信时，也应绑定最近可用微信目标或要求用户先连接微信。

3. 多会话后台执行
   - 允许用户在某个会话生成中自由切换到其他会话。
   - 每个会话维护独立运行状态，不再用单个全局 automation/execution 状态阻塞全部对话。
   - 多个 schedule 触发时应进入各自的计划执行会话，至少独立排队，避免互相覆盖。

4. 高频任务执行模型
   - 分钟级任务优先执行稳定脚本，不应每次都唤起大模型。
   - 只有脚本失败、异常需要解释、或需要生成新报告时再进入 Agent 流程。
   - 后续成功执行后沉淀固定脚本/方法，减少重复试错。

5. UI 与历史一致性
   - 后台 schedule 更新的会话记录需要落盘为第一事实源。
   - 前台切换会话时从真实历史文件刷新，避免内存态、预览态和导出内容不一致。
