# Agent Qt 自动化设计笔记

这份文档记录 Agent Qt 里比较关键的 Agent 机制，方便以后回看为什么要这样设计。它不是用户手册，而是架构备忘：重点是上下文如何流动、执行结果如何被压缩、后台终端如何被模型重新观察，以及哪些地方仍需要继续加固。

## 核心目标

Agent Qt 的目标不是把本地程序伪装成一个完整工具调用平台，而是让大模型在普通 Markdown 对话里稳定地产生可执行指令，再由本地程序负责解析、执行、观察和回传摘要。

整条链路追求三个性质：

- **可执行**：模型输出一个命令块，本地按顺序执行，减少人工复制多段命令的错误。
- **可观察**：执行日志、文件变更、后台终端状态都能回到后续上下文。
- **可压缩**：高价值摘要优先保留，低价值长日志可按标记优先清理。

## 占位符协议

模型输出遵循“一个命令块 + 后续代码块”的协议。命令块里用带编号占位符引用大段文件内容，例如：

```bash
cat > server.py << 'PYEOF'
<!-- Python block 1 -->
PYEOF
```

后续再给出对应的 fenced code block：

```python
# Python block 1
print("hello")
```

执行前，Agent Qt 会先缓存所有代码块，再替换命令块里的编号占位符。这样做的原因是：

- Markdown 渲染更清晰，用户能直接阅读长代码。
- 命令块不需要塞大量 heredoc 内容，模型更不容易把 shell 语法写乱。
- 同一个代码块可以被多个占位符复用。
- 不支持无编号占位符，避免“第几个块”被模型或解析器猜错。

一个关键细节是安全检查只应匹配真正的占位符行。类似 `# Python block 1: Flask 后端 server.py` 是代码注释，不应该被当成占位符。

## 自动化循环

自动化模式下，一轮循环大致是：

1. 用户输入目标。
2. Agent Qt 构造系统提示词、历史上下文和当前指令。
3. 模型输出 Markdown。
4. Agent Qt 解析命令块和占位符。
5. 本地执行命令。
6. 记录执行日志、文件变更、终端摘要。
7. 如果任务未完成，使用这些结果继续请求下一轮模型。

模型完成任务时应回复 `FINAL:` 标记和简短总结，不再输出命令块。

## 文件变更上下文

执行后，Agent Qt 会捕获项目文件变更并生成变更卡片。UI 可以展示完整 diff，但自动化上下文不应该长期携带完整 diff，因为它很容易撑爆上下文。

当前上下文里的文件变更是摘要优先：

- 变更文件名。
- added / deleted 行数。
- added / modified / deleted / binary 状态。
- diff hunk header，用于定位变更区域。
- 内部 git 快照 repo、before commit、after commit。

这意味着模型如果需要完整细节，应自主使用可用的 shell/git/文件读取能力查询，而不是要求用户重新粘贴 diff。

内部 git 是 Agent Qt 缓存目录里的影子仓库，不污染用户项目。它用于：

- 比较执行前后变化。
- 生成 diff 摘要。
- 给模型一个可定向查询的 commit 坐标。
- 支撑 UI 的 undo/redo 校验。

## 低价值上下文块

长日志、终端输出这类内容很重要，但密度低。Agent Qt 用显式边界标记它们：

```text
<<<AGENT_QT_LOW_VALUE_CONTEXT_START kind=execution_log>>>
...
<<<AGENT_QT_LOW_VALUE_CONTEXT_END>>>
```

压缩历史时，这些块可以优先替换成摘要提示；近期上下文里则可以保留原文，帮助模型判断刚发生的错误。

这个设计的核心是区分两类信息：

- **高密度信息**：文件名、增删行数、hunk、terminal id、pid、log path、registry path。
- **低密度信息**：完整 stdout/stderr、长安装日志、构建滚动输出。

高密度信息应该尽量保留；低密度信息按需读取或压缩。

## 后台终端机制

命令分为短命令和后台终端两类。

短命令由执行线程同步运行。常见服务命令会被识别为长运行任务，直接进入底部终端。普通命令如果 30 秒未完成，也会被转入底部终端。

进入底部终端后，Agent Qt 会为每个终端维护：

- terminal id。
- PID。
- cwd。
- command。
- running / exited 状态。
- expected_persistent 标记。
- registry path。
- log path。

这些信息会写入项目缓存目录下的终端注册表：

```text
<project-cache>/terminals/registry.json
```

每个终端还有独立日志文件，持续追加输出：

```text
<project-cache>/terminals/<terminal-id>-<name>.log
```

上下文里不会塞完整持续日志，只保留类似 file change 的终端索引摘要：

```text
Terminal processes:
 - id=... pid=... status=running persistent=True 启动命令=npm run dev
```

模型下一轮只用这一行读取 app 内存里的终端缓冲区：

```bash
curl -s 'http://127.0.0.1:8798/terminallogs?pid=12345'
```

把 `pid=` 后面的值替换成终端摘要里的实际 pid。

## 超时转后台后的模型行为

安装、构建、数据拉取这类命令不一定是常驻服务，但可能超过 30 秒。它们超时转后台后，模型不应把“已进入后台”视为任务完成。

提示词层面的策略是：

1. 如果看到安装、构建、拉取数据、长耗时检查被转入后台，先读取 terminal summary。
2. 如果 registry/log 显示仍在正常运行，输出短暂等待命令。
3. 等待后读取对应 log/registry。
4. 下一轮根据日志判断继续等待、修复错误、验证结果或结束。

这相当于把阻塞等待改成模型可控的轮询循环。UI 不会卡住，模型也不会盲目前进。

## 技能机制

Skill 先做手动选择，不做模型自主发现。用户选择的 `SKILL.md` 会拼接进本轮系统上下文，作为明确约束。

当前策略是保守的：

- 技能存在当前工作区缓存。
- 用户手动选择后才生效。
- Prompt 明确要求模型不要自行寻找或猜测其他技能。

这样避免模型过早引入不可控的“自动技能检索”，后续如果要做自主发现，可以在已有手动机制稳定后扩展。

## 上下文压缩策略

自动化上下文分三段：

1. 系统提示词。
2. 历史对话。
3. 当前指令。

历史会根据预算分层：

- 近期历史保留更完整。
- 较早历史压缩。
- 低价值日志块优先省略。
- 文件变更摘要、终端摘要、错误关键词优先保留。

这套策略的目标不是完美记住一切，而是让模型保留继续工作的坐标：哪里改了、哪个终端在跑、日志去哪看、下一步该验证什么。

## 已加固的关键边界

这些边界曾经是自动化循环里最容易误判的地方，现在已经在机制上补强：

- **观察命令不再误判后台**：读取日志的 `sleep/tail/cat/grep/ps/rg` 等观察命令优先判定为短命令，避免因为日志路径或文本里包含 `npm run dev`、`uvicorn` 等字符串而再次进入后台。
- **终端 registry 保留 completed 历史**：registry 分为 active 和 completed，同时兼容 `processes` 汇总字段。已完成或被关闭的终端会保留最近若干条，方便模型回查日志。
- **超时命令不再重启**：普通命令 30 秒超时后，原 subprocess 会继续在后台运行，stdout/stderr 继续追加到日志文件；UI 会创建一个外部进程观察卡，而不是重新执行同一命令。
- **后台原因结构化**：终端摘要包含 `launch_reason=long_running_pattern|timeout` 和 `command_kind=server|install|build|data_fetch|observe|unknown`，帮助模型判断是继续等待、读取日志，还是进入下一步。
- **压缩保留终端关键词**：`Terminal processes:`、`registry=`、`log=`、`pid=`、`launch_reason=`、`command_kind=` 被视为高信号内容，降低历史压缩后丢失后台任务坐标的概率。

## 仍需观察的点

这套机制仍值得在真实任务中继续观察：

- 外部进程观察卡基于 PID 和日志文件追踪，跨平台进程树关闭在复杂 shell 场景下可能还需要继续打磨。
- 终端日志持续追加，后续可以考虑日志轮转或按大小截断，避免缓存目录无限增长。
- `command_kind` 是启发式分类，真实世界命令很多，后续可根据失败案例继续补规则。

## 设计原则

这套 Agent 机制的关键不是让模型“永远知道所有细节”，而是让它永远知道如何恢复细节：

- 文件变更不塞完整 diff，但给 git 坐标。
- 终端不塞完整日志，但给 registry/log path/PID。
- 长日志可压缩，但错误和索引摘要要保留。
- 后台任务不阻塞 UI，但让模型用 sleep + log polling 自主等待。

只要这些坐标稳定，模型就能像一个本地工程助手一样持续推进，而不是每轮都从一团不可观察的输出里猜。
